# 知库 Claude Skills 复用手册（v1）

## 1. 文档目的

本文档用于把 `Jeffallan/claude-skills` 中适合 `知库` 的方法，压缩成后续开发可直接执行的工作手册。

它不是对原项目的照抄，而是面向 `知库` 当前阶段的落地版本。

---

## 2. 这次提炼出的 5 个核心做法

### 2.1 先建立 Common Ground，再开始写代码

每次开始开发前，先明确：

- 本次目标是什么
- 本次不做什么
- 当前已有能力能否复用
- 用户真正能感知到的结果是什么

如果这 4 个问题没有先说清，就不进入实现阶段。

### 2.2 任务必须走“专家路由”

后续开发默认先给任务归类，再决定实现方式。

`知库` 当前固定的专家路由如下：

- `产品 / 交互`：页面结构、主闭环、文案、状态反馈
- `React / 前端`：组件、状态、交互、视觉层级
- `FastAPI / Python`：接口、服务层、配置、异常处理
- `RAG / 检索`：切块、召回、重排、引用、回退
- `Debug / 排障`：解析失败、接口异常、构建问题
- `Security / 安全`：Cookie、密钥、外部接口、输入校验

### 2.3 复杂任务必须分阶段推进

默认阶段固定为：

1. 对齐目标
2. 拆解链路
3. 实现主路径
4. 验证关键路径
5. 沉淀文档

禁止跳过验证直接进入下一轮需求。

### 2.4 交付物优先，不靠口头共识

后续重要工作必须沉淀为可复用文档，而不是只停留在对话里。

`知库` 当前固定的文档分层：

- `docs/product/`：产品定位、MVP 边界、进度同步
- `docs/engineering/`：开发规范、公共共识、工作手册
- `docs/architecture/`：架构、运行方式、技术说明

### 2.5 每次开发都要带验证点

后续开发默认最少要说明：

- 改了哪些文件
- 用户能看到什么变化
- 用什么方式验证
- 还有什么风险未解除

## 3. 在知库中的固定执行方式

## 3.1 页面类任务

页面类任务固定顺序：

1. 先定义页面在主闭环里的位置
2. 再定义状态：默认 / 加载 / 成功 / 失败 / 空状态
3. 再定义关键动作反馈
4. 最后再补动效与视觉层级

禁止直接从视觉开始倒推产品逻辑。

## 3.2 接口类任务

接口类任务固定顺序：

1. 明确输入输出
2. 下沉服务逻辑
3. 补错误处理与超时
4. 保证配置可注入
5. 做最小验证

## 3.3 RAG 类任务

RAG 类任务固定顺序：

1. 明确回答要基于什么证据
2. 明确怎么切块
3. 明确怎么召回
4. 明确引用怎么返回
5. 明确证据弱时怎么降级

如果证据弱，就不能把答案包装成确定结论。

---

## 4. 后续开发的统一输入模板

后续每次进入一个非小修任务，默认先按下面 6 项对齐：

- `任务目标`
- `本次不做`
- `影响文件`
- `主链路`
- `验证方式`
- `风险与回退`

这 6 项可以简写，但不能缺失。

---

## 5. 当前对知库最有价值的吸收点

结合 `claude-skills` 和 `知库` 当前阶段，最值得长期复用的不是“技能数量”，而是下面三件事：

- 用 `Common Ground` 防止方向漂移
- 用 `专家路由` 防止所有问题都用一种思维处理
- 用 `阶段验证` 防止原型不断堆功能却不稳

这三点，从现在开始作为 `知库` 的默认开发方法。

---

## 6. 从现在开始的执行约束

后续开发默认遵循以下约束：

- 不再直接堆页面，优先补主闭环
- 不再只谈想法，优先产出可复用文档和代码
- 不再给用户暴露调试思维，界面必须保持产品化
- 不再让回答“像模型自由发挥”，优先做证据约束
- 不再用一次性修补方式处理主链路问题
