# 知库 OpenViking 复用手册（v1）

## 1. 文档目的

本文档用于提炼 `volcengine/OpenViking` 中适合 `知库` 的设计思想，并明确后续哪些点直接复用，哪些点只保留方向、不在当前阶段硬上。

---

## 2. OpenViking 对知库最有价值的 5 个知识点

## 2.1 不把知识库只当“向量切片堆”

OpenViking 的核心观点之一，是上下文不能只做扁平切片存储，而要有清晰的组织结构。

这对 `知库` 很重要。

后续 `知库` 的知识库组织，不再只看：

- 一条内容
- 一组 chunks

而要逐步补成：

- 内容对象
- 内容摘要层
- 内容结构层
- 原始正文层
- 会话沉淀层

也就是：**知识库不是一堆片段，而是一套有层级的内容系统。**

## 2.2 分层上下文按需加载

OpenViking 提出的 `L0 / L1 / L2` 分层非常适合 `知库`。

后续可以映射为：

- `L0`：一句话摘要 / 卡片摘要
- `L1`：结构化笔记 / 关键要点 / 小节摘要
- `L2`：原始正文 / 字幕 / 转写全文

这样做的价值很直接：

- 减少问答时一次性塞给模型的噪声
- 让问答先看摘要层，再决定是否下钻
- 为后续做“低成本问答”和“快速预览”打基础

## 2.3 检索不只看语义相似，还要看内容所在结构

OpenViking 的目录递归检索思路，适合转化成 `知库` 的“分层检索”。

在 `知库` 中，不一定要做完整文件系统范式，但可以直接吸收这条原则：

**先定位最相关内容，再在该内容内细找最相关片段。**

也就是后续检索策略优先顺序：

1. 先召回相关内容
2. 再对命中内容内部片段二次检索
3. 必要时结合全库候选做融合

这样比“直接全库平铺 chunk 检索”更贴合真实知识场景。

## 2.4 检索链路必须可观测

OpenViking 很强调“可视化检索轨迹”。

这点非常适合 `知库` 当前阶段。

后续 `知库` 默认要逐步补出：

- 这次先命中了哪些内容
- 为什么命中这些内容
- 最终引用来自哪几个片段
- 这次答案属于全库回答、单文回答还是片段回答

也就是：**用户不仅要看到答案，还要看到答案是怎么来的。**

## 2.5 会话要能沉淀为长期记忆

OpenViking 的“自动会话管理 + 长期记忆提取”很适合做为 `知库` 的中期方向。

对 `知库` 来说，可以转化为：

- 把高价值问答沉淀为知识卡片
- 把用户反复追问的偏好沉淀为检索偏好
- 把某些稳定结论沉淀为“复盘层”

当前我们已经有：

- 会话记录
- 保存为知识卡片

后续可以继续补：

- 问答后的自动归档摘要
- 会话主题自动重命名
- 高频追问主题聚类

---

## 3. 在知库中的直接落地

## 3.1 现在就该采用的

- 内容三层表示：摘要层 / 结构层 / 原文层
- 检索先找内容、再找片段
- 问答页面展示检索依据与来源路径
- 会话结果可沉淀为知识卡片

## 3.2 下一阶段适合补的

- 内容层级索引
- 单条内容内二次召回
- 会话自动压缩摘要
- 回答依据轨迹更细粒度展示

## 3.3 当前不硬上的

以下内容暂时不直接照搬：

- 独立上下文数据库服务
- 完整虚拟文件系统协议
- 重度 VLM 驱动的多模态解析体系

原因很简单：

- 当前 `知库` 还在 MVP 强化阶段
- 先把 B站 / 网页 / 问答主链路做稳更重要
- 现在更适合做“思想吸收 + 轻量落地”，而不是整体重构

---

## 4. 对知库架构的具体启发

## 4.1 内容模型启发

后续 `ContentDetail` 可以长期演化为：

- 基础元信息
- L0 摘要
- L1 结构化笔记
- L2 原始正文
- chunks
- 来源引用
- 问答沉淀

## 4.2 检索模型启发

后续检索推荐升级顺序：

1. 全库内容级召回
2. 内容内片段级召回
3. 多路融合
4. 证据强度评估
5. 检索轨迹展示

## 4.3 产品体验启发

问答页后续要继续朝下面方向优化：

- 不是只给答案
- 而是给“答案 + 依据 + 来源路径 + 下一步动作”

详情页后续要继续朝下面方向优化：

- 不是只展示正文
- 而是展示“这条内容如何被整理、如何被命中、如何被引用”

---

## 5. 对当前知库最有帮助的新增约束

### 约束 1：整理链路必须产出分层结果

后续每次导入内容，目标不是只拿到一段正文，而是尽量产出：

- 摘要
- 要点
- 结构化笔记
- 切块

### 约束 2：检索链路优先做层级化，而不是继续平铺化

后续检索优化优先做：

- 内容级候选
- 内容内精排

而不是只继续堆全库 chunk 检索变体。

### 约束 3：问答必须保留证据感

后续问答输出默认要越来越清楚地表达：

- 回答基于哪些资料
- 命中强还是弱
- 哪些片段最值得回看

### 约束 4：会话不是临时聊天，应尽量可沉淀

后续会话系统默认往“可复盘、可沉淀、可再利用”方向发展。

## 6. 知库后续可直接采用的实施项

结合当前仓库状态，最适合吸收 OpenViking 的顺序是：

### P0

- 把内容详情页的数据分层展示得更清楚
- 把检索结果展示从“命中数”升级到“命中路径”

### P1

- 增加“先内容、后片段”的二段式召回
- 增加会话自动摘要

### P2

- 增加知识卡片和来源内容之间的关系视图
- 增加长期记忆/高频主题归档

---

## 7. 与当前方法论的关系

结合当前已吸收的方法：

- `BiliNote`：内容采集与整理
- `ragent`：问答策略与检索增强
- `claude-skills`：Common Ground 与专家路由
- `superpowers`：设计、计划、调试、验证纪律
- `OpenViking`：分层上下文、层级检索、可观测、会话沉淀

从现在开始，`知库` 的方向会更明确：

**不是做一个“能聊天的导入器”，而是做一个“能采集、能整理、能追问、能沉淀”的个人知识系统。**
