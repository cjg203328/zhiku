# 知库全栈工程技能基线（v1）

## 1. 文档目的

本文档基于 `Jeffallan/claude-skills` 的方法论进行提炼，目标不是照搬插件体系，而是把其中对 `知库` 最有价值的工程能力沉淀为可复用规范。

从本文档起，后续开发默认按这套方法执行。

---

## 2. 我们吸收的核心思想

`claude-skills` 对我们最有价值的，不是“技能数量”，而是这四个工程原则：

### 2.1 专家路由

不同任务要切换不同思维模式，而不是始终用一种方式写代码。

在 `知库` 中，后续开发默认切换以下 5 种专家视角：

- `FastAPI / Python` 视角
- `React / 前端产品` 视角
- `RAG / 检索架构` 视角
- `Debugging / 排障` 视角
- `Security / 安全` 视角

### 2.2 分阶段工作流

复杂任务不能直接写代码，必须按阶段推进：

- 发现问题
- 规划方案
- 执行实现
- 验证结果
- 复盘沉淀

### 2.3 显式假设

任何开发都建立在假设上。

如果假设不被显式记录，后续就会出现：

- 做偏
- 重复返工
- UI 和后端目标不一致
- 技术选型摇摆

### 2.4 约束优先

高质量开发不是“功能越多越好”，而是：

- 明确必须做什么
- 明确绝不能做什么
- 明确每步的验证点

---

## 3. 知库默认工作流

后续默认采用如下工作流：

### 阶段 A：Common Ground

开始开发前，先确认：

- 当前要解决的是哪一层问题
- 当前版本的真实目标是什么
- 本次改动不做什么
- 当前已有实现是否足够复用

### 阶段 B：规划

规划时必须明确：

- 改哪些文件
- 哪个链路会被影响
- 怎么验证
- 风险在哪里

### 阶段 C：实现

实现时必须：

- 优先修主链路
- 优先做可复用能力
- 避免表层补丁
- 不改与任务无关的问题

### 阶段 D：验证

至少验证：

- 后端编译
- 前端构建
- 与本次功能直接相关的主路径

### 阶段 E：同步

每次交付都要同步：

- 改了什么
- 用户能看到什么
- 验证了什么
- 下一步是什么

## 4. FastAPI / Python 基线

这部分主要吸收 `FastAPI Expert` 的方法。

### 4.1 默认工作流

1. 先分析接口需求
2. 再设计请求/响应模型
3. 再实现服务层与路由层
4. 再补安全与错误处理
5. 最后验证接口行为

### 4.2 必须做

- 所有 I/O 优先异步思维
- 类型标注完整
- 请求 / 响应模型清晰
- 路由层尽量轻，复杂逻辑下沉到服务层 / 仓储层
- 错误信息用户可理解
- 返回结构可扩展

### 4.3 禁止做

- 把大段业务逻辑塞进 router
- 混乱的同步 / 异步调用
- 不经校验直接信任外部输入
- 硬编码敏感配置
- 只返回 `500` 不给可读错误

### 4.4 在知库里的落地

后端新增功能默认按这个结构分层：

- `routers/`：接口入口
- `services/`：业务流程
- `repositories/`：SQLite / 检索 / 持久化

---

## 5. React / 前端基线

这部分主要吸收 `React Expert` 的方法。

### 5.1 默认工作流

1. 先分析页面目标和状态流
2. 选择合适的数据获取与状态管理方式
3. 用明确类型实现组件
4. 先保证类型通过
5. 再优化交互与层级
6. 再做体验打磨

### 5.2 必须做

- 组件状态职责清晰
- 异常、空状态、加载状态都要可读
- 文案是产品文案，不是工程文案
- 重要操作必须有反馈
- 交互链路必须围绕主闭环

### 5.3 禁止做

- 只做“能显示”的页面，不做状态兜底
- 出现调试页感
- 把用户带进技术细节
- 让用户面对“静默失败”

### 5.4 在知库里的落地

前端所有页面默认遵循：

- 首页：产品入口，不是状态面板
- 导入页：低成本输入，失败可理解
- 详情页：便于阅读、追问、重解析
- 问答页：可回看、可引用、可沉淀

---

## 6. RAG / 检索架构基线

这部分主要吸收 `RAG Architect` 的方法。

### 6.1 RAG 默认工作流

1. 先明确检索目标
2. 设计切块策略
3. 设计召回策略
4. 设计重排策略
5. 建立评估与迭代机制

### 6.2 在知库中的当前执行顺序

`知库` 的 RAG 升级顺序固定为：

1. 内容切块
2. 问题改写
3. 多路召回
4. 融合排序
5. 上下文生成回答
6. 质量评估与回退

### 6.3 必须做

- 每次回答都尽量带引用
- 检索链路可解释
- 数据源元信息可追溯
- 先做混合检索，再考虑复杂向量方案
- 检索弱时要降低回答自信度

### 6.4 禁止做

- 只靠模型自由发挥
- 检索命中弱时还给出高确定性结论
- 不记录召回策略和质量状态

### 6.5 在知库里的落地

当前后续开发默认按 `ragent-lite` 方向推进：

- 问题改写
- 标题/摘要/正文/标签/片段多路召回
- 会话上下文参与检索
- 后续继续补重排与质量控制

---

## 7. Debugging 基线

这部分主要吸收 `Debugging Wizard` 的方法。

### 7.1 默认排障流程

1. 先复现
2. 再隔离
3. 一次只验证一个假设
4. 修复后回归验证
5. 补防回归措施

### 7.2 必须做

- 先看真实报错
- 先判断是不是环境问题
- 不要连续盲改多个点
- 修复后保留最小验证路径

### 7.3 在知库里的落地

遇到导入失败、问答失败、构建失败时，默认顺序：

- 看错误文本
- 看日志 / 编译结果
- 看请求参数
- 看最近改动文件
- 做单点修复

---

## 8. Security 基线

这部分主要吸收 `Secure Code Guardian` 的方法。

### 8.1 必须做

- 所有输入都默认不可信
- 不在代码中硬编码密钥
- 不把敏感信息写进用户可见错误
- 不把秘密写进仓库
- 配置类能力统一走环境变量或本地配置

### 8.2 在知库里的落地

尤其要注意：

- 第三方模型 `API Key`
- B站 Cookie
- 日志中的敏感信息
- 导入外链正文抓取的输入边界

## 10. 后续开发强制执行规则

### 规则 1

每次开发前，先确认本次任务属于哪一层：

- 采集层
- 整理层
- 检索层
- 问答层
- 产品体验层

### 规则 2

每次开发时，必须显式说明本次采用哪种专家视角。

### 规则 3

每次开发完成后，必须至少给出：

- 修改内容
- 验证结果
- 下一步建议

### 规则 4

对复杂改动，优先按“发现 -> 规划 -> 实现 -> 验证 -> 同步”五步执行。

### 规则 5

如果某个实现会降低产品主闭环清晰度，哪怕技术上可做，也要降级优先级。

---

## 11. 本文档的实际作用

这不是一个“技能收藏文档”，而是 `知库` 的默认工程基线。

从现在开始，后续开发默认遵循：

- 显式假设
- 专家路由
- 分阶段工作流
- 可验证交付
- 面向主闭环优化

这就是我们从 `claude-skills` 中真正复用到 `知库` 的部分。
