# 知库协作与进度同步规范（v1）

## 1. 文档目的

这份文档用于统一 `知库` 项目的日常协作方式，重点解决两个问题：

- 每次修改后，都要有一份能直接回看的实时进度文档
- 远程仓库推送权保留给用户手动执行，不默认自动推送

---

## 2. 角色分工

### 2.1 代理 / 开发执行方

- 负责分析、实现、验证、本地整理
- 负责实时更新本轮变更记录
- 负责把“当前是否适合提交 / 推送”说清楚
- 默认不直接执行 `git push`

### 2.2 用户

- 负责确认方向、验收结果、决定提交说明
- 负责最终的 `git commit` / `git push`
- 如果需要代理代为推送，必须明确提出

---

## 3. 必须遵守的工作流

### 3.1 开始改动前

先确认本轮任务属于哪一类：

- 采集层
- 整理层
- 检索层
- 问答层
- 产品体验层
- 工程流程 / 基础设施

如果本轮会落地代码、脚本、配置或正式文档，默认要更新：

- `docs/product/知库_开发变更日志.md`

### 3.2 实施过程中

每当以下任一情况发生时，要同步刷新变更日志：

- 改动范围变化
- 新增关键文件
- 验证结果出来
- 发现新的遗留风险
- 本轮目标从“进行中”变成“已完成 / 阻塞”

### 3.3 交付前

交付前必须同时完成这几件事：

1. 更新 `docs/product/知库_开发变更日志.md`
2. 写清楚验证结果
3. 明确当前遗留
4. 提醒用户用手动方式完成提交与推送

---

## 4. 文档分层

### 4.1 高层进度文档

文件：

- `docs/product/知库_开发进度同步.md`

用途：

- 记录项目阶段判断
- 记录已完成能力、当前不足、优先级和阶段路线
- 不要求每个小改动都改这份

### 4.2 实时变更日志

文件：

- `docs/product/知库_开发变更日志.md`

用途：

- 记录每一轮实际落地修改
- 作为“这轮能不能提交”的直接依据
- 必须保持最新记录在最前

---

## 5. 单次变更的最小记录格式

每一轮改动至少记录以下内容：

- 日期
- 任务标题
- 状态
- 任务类型
- 本次完成
- 改动文件
- 本次验证
- 当前遗留
- 下一步

---

## 6. 手动推送规则

默认推送流程如下：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\dev\pre_push_check.ps1
git add -A
git commit -m "你的提交说明"
git push origin main
```

说明：

- `pre_push_check.ps1` 用于检查当前改动里是否同步了进度文档
- 代理默认可以帮你把仓库整理到“可提交”状态
- `git push` 默认由你自己手动执行

---

## 7. 例外规则

以下情况可以不强制新增一条变更日志：

- 纯只读调研
- 未落地任何文件修改
- 仅临时排查但没有形成最终变更

但只要进入“改了文件并准备交付”的状态，就必须补齐变更日志。

---

## 8. 当前仓库约束总结

从本规范起，`知库` 仓库默认采用以下协作模式：

- 修改必须可追踪
- 进度必须落文档
- 验证必须写结果
- 推送必须由用户手动决定
