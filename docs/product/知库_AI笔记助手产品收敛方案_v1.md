# 知库 AI 笔记助手产品收敛方案（v1）

## 1. 文档目的

本文档用于把当前 `知库` 的产品方向进一步收敛为：

**一个以 B站优先跑通、可逐步扩展到网页和本地数据源的 AI 笔记助手。**

它回答 5 个关键问题：

- 产品的真正核心是什么
- 第一阶段必须先跑通哪条链路
- 视频内容应如何完整获取并组织
- 笔记为什么要分成两类
- UI 和后续问答系统应如何围绕这条主链路设计

---

## 2. 产品重新定义

### 2.1 一句话定义

`知库` 不是泛知识库，也不是单纯的视频解析工具。

`知库` 的第一性目标应该定义为：

**把用户给出的链接或文件，自动转成可阅读、可检索、可追问、可沉淀的 AI 笔记。**

### 2.2 当前产品核心

当前最重要的不是“支持很多平台”，而是先把一个最有价值、最容易验证的闭环跑透：

`B站分享链接 -> 完整内容提取 -> 自动建档立库 -> 生成双层笔记 -> 语义检索 -> 对话问答`

### 2.3 长期支持的数据源

长期支持范围可以包括：

- 视频内容
- 网页 / 文章链接
- 通用 URL
- 本地文件
- `TXT / Markdown / Word / PDF`
- 数据表 / 结构化表格数据

但当前阶段必须先明确优先级：

### P0 当前唯一核心

- `B站分享链接`

### P1 第二阶段扩展

- 网页 / 文章 URL
- 本地文档文件

### P2 第三阶段扩展

- 数据表
- 更复杂的多源混合资料

---

## 3. 第一阶段必须成立的用户价值

第一阶段用户真正要的不是“技术能力展示”，而是下面这个体验：

1. 用户只贴一个 `B站分享链接`
2. 平台自动识别视频
3. 平台自动尽可能拿到完整可用内容
4. 平台自动把内容整理成知识笔记
5. 用户可以立即阅读、搜索、提问
6. 回答能够回到原始证据
7. 用户可以把高价值结果继续沉淀成平台知识资产

这意味着：

- 用户不应该先理解复杂配置
- 用户不应该手动做字幕整理
- 用户不应该自己切片、归档、建标题
- 用户不应该只得到一大段原文堆砌

---

## 4. B站优先链路的产品目标

## 4.1 P0 硬目标

P0 的目标不是“完美支持所有 B站内容”，而是先稳定支持：

- 公开可访问的视频
- 用户通过分享获得的标准链接
- 单个视频的标题、简介、字幕或转写正文
- 自动生成可读笔记
- 基于笔记继续问答

### P0 成功标准

只要满足以下条件，就算当前阶段成立：

1. 用户只需要粘贴分享链接
2. 系统能自动识别并归一化链接
3. 系统能获取标题、作者、封面、简介、标签、正文来源
4. 系统能生成带时间戳的文本片段
5. 系统能自动生成一份精炼笔记
6. 系统能把原始文本和精炼结果一起入库
7. 用户可以按语义提问，并看到回答依据
8. 命中片段后，系统可以跳回原视频对应时间位置

---

## 5. 对“视频 -> 时间戳文字 -> 向量 -> 跳转播放”的优化建议

你的方向是对的，而且应该作为 `知库` 的视频笔记主线路。

但如果只停留在：

`视频 -> 转文字 -> 向量化 -> 命中 -> 跳转`

还不够完整。

更适合 `知库` 的结构应该升级为：

`视频 -> 多源内容获取 -> 带时间戳的结构化转写 -> 双层笔记生成 -> 分层索引 -> 语义检索 -> 时间戳回跳`

### 5.1 为什么要升级成这个结构

因为用户要的不是“拿到转写稿”，而是：

- 看得懂
- 找得到
- 问得准
- 能回原文
- 能持续沉淀

### 5.2 更优的数据组织方式

视频内容至少应该拆成三层：

#### 第一层：原始内容层

保留证据与来源，不做过度美化：

- 原始链接
- 标题 / 作者 / 发布时间 / 标签
- 简介
- 字幕正文
- ASR 转写正文
- 每段文本的开始时间和结束时间
- 内容来源标记：字幕 / ASR / 页面简介 / 用户补充

#### 第二层：精炼笔记层

面向阅读和吸收，不是面向存档：

- 一句话总结
- 3 到 7 条核心观点
- 主题分段
- 结构化提纲
- 可复用的知识卡片
- 对用户真正有价值的信息提炼

#### 第三层：检索索引层

面向问答和回溯：

- 分段 chunk
- 语义向量
- 倒排检索字段
- 内容层与片段层的映射关系
- 时间戳定位信息

### 5.3 为什么笔记必须分两类

后续每条视频笔记都应该明确区分为：

#### A. 精炼笔记

用于：

- 快速阅读
- 理解重点
- 生成知识卡片
- 高层问答

#### B. 原始转化内容

用于：

- 完整留档
- 精确检索
- 证据回溯
- 时间戳跳转

两者不是二选一，而是必须并存。

如果只有精炼笔记，证据链会变弱。

如果只有原始转写，阅读体验会很差。

---

## 6. B站内容获取策略

“获取链接内的所有内容”在产品上不能理解成“抓取一切可抓数据”，而应该理解成：

**尽最大可能获取对用户理解视频内容有价值的全部有效信息。**

### 6.1 当前建议保留的高价值内容

- 标题
- 作者
- 封面
- 简介
- 标签 / 分区
- 发布时间
- 字幕正文
- 无字幕时的音频转写结果
- 分段时间戳文本
- 可选的章节结构

### 6.2 当前不建议作为 P0 核心的内容

以下内容可以保留为后续增强，但不应阻塞 P0：

- 全量评论抓取
- 弹幕全量入库
- 视频文件本体下载
- 复杂 OCR 视频画面识别

原因：

- 成本高
- 噪声大
- 很容易偏离“AI 笔记助手”的主价值

---

## 7. 推荐的 B站数据处理 Pipeline

后续 `知库` 的 B站主链路建议统一定义为：

1. 接收分享链接
2. 归一化链接，识别 `BV / 短链 / 长链 / 分P`
3. 获取基础元数据
4. 优先抓公开字幕
5. 无字幕时回退到音频转写
6. 将文本整理成带时间戳的分段 transcript
7. 生成精炼笔记
8. 建立内容层和片段层索引
9. 入库并生成详情页档案
10. 向问答系统暴露可检索数据

### 7.1 transcript 片段最少字段

后续每个时间片段建议最少保存：

- `content_id`
- `segment_id`
- `start_ms`
- `end_ms`
- `text`
- `summary`
- `source_kind`
- `quality_level`
- `embedding_status`

### 7.2 精炼笔记最少字段

- `title`
- `summary`
- `key_points`
- `outline`
- `highlight_quotes`
- `note_markdown`
- `derived_from_content_id`

---

## 8. 问答与 RAG 设计原则

后续问答不应该只基于“原始大段文本”，而要优先走分层检索：

1. 先命中精炼笔记层
2. 再命中原始 transcript 片段层
3. 最终把高层总结和底层证据一起提供给模型

### 8.1 推荐的问答策略

- 高层理解型问题：优先使用精炼笔记
- 细节确认型问题：优先使用时间戳片段
- 证据不足时：显式降级表达
- 回答结果必须保留引用
- 引用最好能直接映射到时间戳

### 8.2 时间戳跳转能力

后续问答或检索命中后，产品层应支持：

- 显示命中片段的时间范围
- 点击后跳回视频对应位置
- 在详情页里高亮对应片段

这样用户获得的是：

`回答 -> 证据 -> 原视频定位`

而不是只有一段“像是对的文字”。

## 9. UI 与主题方向

你给的参考图非常明确，说明你要的不是传统后台，也不是花哨玻璃面板，而是：

- 清晰
- 极简
- 高级
- 内容优先
- 空间感强

### 9.1 UI 关键词

- 大留白
- 强标题层级
- 干净的浅色背景
- 克制的卡片边界
- 明确的标签与分段
- 少而准的强调色

### 9.2 不建议继续强化的方向

以下方向不应成为主视觉：

- 过重玻璃感
- 太多渐变装饰
- 太多颜色竞争
- 明显的调试台 / 后台风
- 为了“AI 感”刻意做夸张动效

### 9.3 推荐的详情页结构

结合你给的参考图，视频笔记详情页建议收敛成：

1. 顶部标题区
2. 来源信息区
3. 标签区
4. Tab 区

Tab 至少保留：

- `链接原文`
- `精炼笔记`
- `原始转写`
- `追加笔记`

### 9.4 推荐的详情页阅读逻辑

默认先进入：

- `精炼笔记`

然后用户可以按需切换到：

- `原始转写`

在 `原始转写` 中要强调：

- 时间戳
- 片段结构
- 命中定位

### 9.5 当前前端改造方向

现有前端已经有产品化基础，但后续视觉应从“玻璃感原型”继续收敛为“内容型阅读产品”：

- 视觉更克制
- 字体层级更强
- 信息密度更精炼
- 卡片数量更少
- 让正文与笔记成为第一视觉中心

---

## 10. 与当前仓库的对接方式

当前仓库不需要推倒重来，但后续改造必须围绕下面几条主线：

### 10.1 采集层

继续强化：

- B站链接识别
- 字幕抓取
- ASR 回退
- 质量状态标记

### 10.2 整理层

补成正式 pipeline：

- 解析
- 清洗
- transcript 分段
- 精炼笔记生成
- 入库

### 10.3 检索层

从当前轻量检索升级为：

- 精炼笔记召回
- transcript 片段召回
- 多路融合
- 时间戳引用

### 10.4 产品体验层

重点改造：

- 导入结果页
- 视频笔记详情页
- 问答页引用展示

---

## 11. 后续开发优先级

### P0：先把 B站视频笔记助手跑通

必须完成：

- 分享链接导入
- 元数据获取
- 字幕优先 / ASR 回退
- 带时间戳 transcript
- 精炼笔记与原始转写双层展示
- 基于片段检索与问答
- 时间戳回跳

### P1：扩展到网页和本地文档

必须完成：

- 网页正文抽取
- 本地文档解析
- 与视频同样的双层笔记结构

### P2：平台数据源与更强问答

必须完成：

- 多源内容混合检索
- 会话记忆沉淀
- 数据表 / 结构化数据源接入

---

## 12. 当前强制约束

从本文件起，后续开发默认新增以下约束：

### 约束 1

当前产品默认名称应理解为：

**AI 笔记助手**

而不是泛化工具箱。

### 约束 2

当前阶段所有新功能，必须优先服务：

`B站分享链接 -> 双层笔记 -> 问答 -> 证据回跳`

### 约束 3

每条视频内容默认要产出两类笔记：

- 精炼笔记
- 原始转写内容

### 约束 4

问答结果默认必须能回到：

- 原文片段
- 时间戳
- 来源内容

### 约束 5

UI 默认朝“极简、高级、内容优先”收敛，不再继续走后台化或花哨化路线。

---

## 13. 当前一句话共识

`知库` 当前最正确的方向不是做一个“能导内容的工具”，而是做一个：

**以 B站优先跑通、可自动建档立库、可生成双层笔记、可继续问答回溯的 AI 笔记助手。**
