# 知库 MVP 与开发规范（v2）

## 1. 文档目的

这份文档用于统一 `知库` 后续开发方向，避免继续出现“功能很多，但主价值不够聚焦”的问题。

本文档只回答四件事：

- 这个产品到底要先解决什么问题
- 首个 MVP 必须做什么，不做什么
- 技术和产品应如何收敛
- 后续开发必须遵循哪些约束

---

## 2. 产品重新定义

### 2.1 一句话定义

`知库` 是一个面向个人学习与研究场景的 **内容采集、知识整理、知识问答工具**。

它的核心不是“聊天”，也不是“视频下载器”，而是：

**把可抓取内容转成可复用知识，再让用户围绕这些知识继续提问、整理、沉淀。**

### 2.2 当前优先场景

当前优先级必须严格收敛为：

1. `B站视频链接导入`
2. `文章 / 网页链接导入`
3. `解析为文本数据源`
4. `入库、归档、命名`
5. `基于知识库做检索与问答`
6. `把问答继续沉淀成知识卡片`

### 2.3 产品本质

这个产品的本质是三层结构：

- **采集层**：从链接或文件拿到可处理内容
- **整理层**：把原始内容变成可读、可检索、可引用的数据
- **问答层**：围绕整理后的知识库做 RAG 检索与智能问答

---

## 3. 当前方向评估

## 3.1 你的方向是对的

你现在提出的方向是成立的，而且适合先做 MVP：

- 先从 `B站` 做起，成本最低、验证最快
- 再扩展到有公开文本结构的文章 / 网页
- 先做“可抓取内容”而不是“全平台全能解析”
- 先做“知识整理 + 问答”而不是“复杂协作产品”

## 3.2 当前项目已有基础

当前仓库已经具备以下基础能力：

- 网页端产品原型
- 本地 API 服务
- B站链接解析基础链路
- 设置页与模型接入基础能力
- 问答页、引用、会话、问答保存卡片
- 内容重解析能力

这意味着当前项目 **不需要推倒重来**，而是应该：

**从“功能演示型原型”升级为“链路清晰的产品型 MVP”。**

## 3.3 当前最大问题

目前最大问题不是页面，而是主链路仍然不够清晰：

- 采集层成功率不稳定
- 整理层缺少统一 pipeline 定义
- 检索层还偏轻量
- 问答层还没有完整采用 RAG 策略
- 产品上还没有形成“导入 -> 整理 -> 提问 -> 沉淀”的强闭环

---

## 4. MVP 核心目标

MVP 不追求“大而全”，只验证一个问题：

**用户能否把一条 B站或网页内容变成自己的知识资产，并基于它高效问答。**

### 4.1 MVP 核心闭环

MVP 必须打通以下闭环：

1. 用户输入一个链接
2. 系统识别来源平台
3. 系统抓取标题、元数据、正文来源
4. 对视频优先抓字幕，失败再走音频转写增强
5. 对文章优先抓正文文本
6. 系统用模型整理成摘要、要点、结构化笔记
7. 系统把内容入库
8. 用户给内容命名、分类、打标签
9. 用户围绕知识库内容进行问答
10. 问答结果可追溯、可沉淀、可继续复用

### 4.2 MVP 成功标准

只要满足以下条件，就算 MVP 成立：

- 用户能成功导入一条公开视频链接
- 用户能看到结构化笔记结果
- 用户能基于这条内容发起问答
- 回答能展示引用来源
- 用户能把回答保存为知识卡片

---

## 5. MVP 功能边界

## 5.1 必须做（Must Have）

### A. 内容导入

- 支持 `B站 BV 号 / 短链 / 长链`
- 支持 `网页 / 文章 URL`
- 导入状态可见：`待处理 / 解析中 / 已完成 / 失败`
- 导入失败必须给用户可理解原因

### B. 内容解析

- 视频：
  - 标题、作者、封面、基础元数据
  - 优先字幕抓取
  - 无字幕时可切音频转写增强
- 网页：
  - 标题
  - 正文文本提取
  - 链接与来源记录

### C. 内容整理

- 自动生成：
  - 一句话摘要
  - 3-5 条要点
  - 结构化笔记
- 内容切块
- 建立全文检索与后续向量检索入口

### D. 知识库管理

- 列表页
- 详情页
- 标签 / 分类 / 命名
- 最近导入
- 回收站
- 单条内容重解析

### E. 问答

- 基于知识库检索回答
- 回答带引用
- 会话历史可回看
- 问答可保存为知识卡片

### F. 设置与运行

- 模型配置
- 导入 / 日志 / 导出目录可见
- B站登录态状态可见
- 手动备份

## 5.2 推荐做（Should Have）

- 问题改写
- 多路检索融合
- 重排
- 模型失败回退
- 网页正文抽取质量评分
- 内容质量标签（字幕 / 转写 / 页面抽取）

## 5.3 暂不做（Not Now）

- 小程序
- 多端同步
- 团队协作
- 分享链接
- 知识图谱可视化
- 抖音 / 小红书高维护解析
- 完整自动化视频 OCR

---

## 6. 数据源策略

## 6.1 优先级顺序

当前数据源策略统一为：

### 视频类

1. 平台公开字幕
2. 页面简介 / 公开文本
3. 音频转写
4. 用户手动补充

### 文章类

1. 页面正文抽取
2. Meta 信息
3. 用户手动补充

## 6.2 重要原则

- 不承诺“所有内容都能解析”
- 只承诺“尽最大可能拿到可用文本”
- 必须记录解析来源与质量状态

每条内容都应该有如下元信息：

- `source_type`
- `platform`
- `source_url`
- `parse_mode`
- `has_subtitle`
- `asr_used`
- `llm_enhanced`
- `quality_level`

---

## 7. RAG 策略方向

参考 `ragent` 的思想，`知库` 后续问答必须逐步升级为标准 RAG 流程，而不是停留在简单关键词检索。

### 7.1 第一阶段：当前可落地策略

- 标题 / 摘要 / 正文 / 片段混合召回
- 中英混合检索词拆分
- 引用回链到内容与片段

### 7.2 第二阶段：推荐尽快实现

- 问题改写
- 多路召回
- 结果重排
- 会话上下文参与检索

### 7.3 第三阶段：增强策略

- 模型路由
- 检索失败回退
- 答案质量评分
- 首包探测与失败熔断

---

## 8. 技术实现约束

## 8.1 总体原则

- 先修主链路，不做炫技堆料
- 先保证可回溯，再提升“聪明度”
- 所有新功能必须优先服务主闭环

## 8.2 后端规范

- 新增能力优先做成服务层和仓储层复用能力
- 不允许把复杂逻辑全塞进 router
- 所有导入链路必须返回可读错误
- 所有重解析必须覆盖更新原内容，避免重复内容
- 所有问答返回必须支持 citations

## 8.3 前端规范

- 页面必须产品化，不出现调试页感
- 所有空状态都要写成用户可理解文案
- 任何失败都不能只显示 `500`
- 重要操作必须给反馈：处理中、成功、失败
- 所有新入口都优先挂在主路径，不做隐藏能力

## 8.4 数据规范

每条内容最少包含：

- 标题
- 摘要
- 正文
- 标签
- 元信息
- 分块结果

每条问答最少包含：

- 问题
- 回答
- 引用
- 会话归属

---

## 9. 后续开发优先级

## P0：先完成产品成立

1. B站解析成功率继续提升
2. 网页正文抽取能力补上
3. 统一内容整理 pipeline
4. 问答结果质量提升

## P1：把问答真正做聪明

1. 问题改写
2. 多路检索融合
3. 重排
4. 会话记忆参与回答

## P2：增强可用性

1. 批量导入
2. 导入队列面板
3. 内容质量标识
4. 解析报告与可视化状态

---

## 10. 开发执行要求

从本文件起，后续开发必须遵守以下规则：

### 规则 1

新功能如果不能强化主闭环，原则上不进当前迭代。

### 规则 2

所有开发任务必须标注属于以下哪一层：

- 采集层
- 整理层
- 检索层
- 问答层
- 产品体验层

### 规则 3

每完成一个功能，必须同步三件事：

- 改了什么
- 验证了什么
- 下一步是什么

### 规则 4

用户看到的内容必须是产品语言，不是工程语言。

### 规则 5

如果某个功能只是“看起来很高级”，但不能明显提升导入成功率、整理质量或问答质量，则优先级降低。

---

## 11. 本文档结论

`知库` 现在最正确的方向不是继续发散，而是收敛成：

**一个以 B站 / 网页内容采集为入口，以知识整理为中层，以 RAG 问答为核心价值的个人知识库产品。**

MVP 的目标不是做全，而是先证明：

**用户真的愿意把内容放进来，并围绕这些内容持续提问和沉淀。**
