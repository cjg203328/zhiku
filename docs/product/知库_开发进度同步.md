# 知库开发进度同步

## 1. 当前阶段结论

当前项目已从“工程骨架”进入“产品型原型”阶段，但还没有完全进入“稳定可验证 MVP”阶段。

目前最重要的任务不是继续铺更多页面，而是继续强化：

- 采集成功率
- 内容整理质量
- 检索与问答质量
- 主闭环完整度

---

## 2. 当前已完成

## 2.1 产品与界面

- 网页端主要页面已完成产品化改造
- 中文文案、层级、玻璃感与卡片风格已统一
- 设置、知识库、内容详情、问答页、引导页已有基础产品形态

## 2.2 内容导入

- B站链接导入基础链路已接通
- 支持 BV 号、短链、长链基础识别
- 导入失败可回退为预览内容，避免页面直接报错

## 2.3 内容整理

- 已支持摘要、要点、结构化笔记
- 已支持内容切块
- 已支持单条内容重解析

## 2.4 模型接入

- 已支持 OpenAI 兼容接口
- 已补国产模型推荐预设
- 设置页保存后可立即生效，并写入本地 `.env`

## 2.5 问答

- 已支持基于知识库片段问答
- 已支持引用来源展示
- 已支持 follow-up 追问建议
- 已支持会话记录保存与回看
- 已支持把问答结果保存为知识卡片

## 2.6 工程规范

- 已提炼 `claude-skills` 方法论，沉淀为 `知库` 可复用工程规范
- 已补 `Common Ground`、工程技能基线、复用手册
- 后续开发默认按“先对齐目标 -> 再实现主链路 -> 再验证 -> 再同步”的方式推进
- 已补充 `superpowers` 方法提炼，后续默认增加“系统化调试”和“验证先于完成宣称”两条硬约束
- 已补充 `OpenViking` 方法提炼，后续默认增加“分层上下文、层级检索、检索可观测、会话沉淀”四条产品与架构约束

---

## 3. 当前不足

## 3.1 采集层不足

- B站解析成功率仍不够稳定
- 网页 / 文章正文抽取还未形成正式能力
- 内容质量状态还不够清晰

## 3.2 整理层不足

- 还没有统一的“解析 -> 清洗 -> 整理 -> 切块 -> 入库” pipeline 定义
- 结构化笔记质量仍受模型与原始文本质量波动影响

## 3.3 检索层不足

- 当前仍以轻量检索为主
- 还缺问题改写、多路召回、重排
- 会话上下文尚未真正进入检索链

## 3.4 问答层不足

- 还没有真正的多轮记忆问答
- 还没有模型失败回退与模型路由
- 回答质量评估能力还未补齐

---

## 4. 下一阶段开发顺序

## P0：主闭环强化

### 目标

让 `导入 -> 整理 -> 问答 -> 沉淀` 成为稳定可演示、可验证、可复用的产品链路。

### 本阶段任务

1. 提升 B站解析成功率
2. 接入网页正文抽取
3. 统一整理 pipeline
4. 补内容质量标识

## P1：问答升级

### 目标

让问答从“能回答”变成“更像知识助理”。

### 本阶段任务

1. 问题改写
2. 多路检索融合
3. 检索重排
4. 会话上下文参与问答

## P2：产品完善

### 目标

提升普通用户理解成本与操作流畅度。

### 本阶段任务

1. 导入队列
2. 内容质量提示
3. 失败排查入口
4. 会话重命名 / 删除 / 置顶

---

## 5. 当前推荐下一步

按当前仓库状态，下一步最该做的是：

### 第一优先级

**网页 / 文章 URL 抽取能力**

原因：

- 这能直接验证“文章等等有链接 api 的内容进行抓取”的产品方向
- 这比继续堆更多页面更有价值
- 这会让产品从“B站单点工具”变成“内容采集型知识库”

### 第二优先级

**问题改写 + 多路检索融合**

原因：

- 这会直接提升用户对“聪明度”的感知
- 这与 `ragent` 的策略方向一致
- 这比单纯换模型更能改善问答体验

### 第三优先级

**多轮问答记忆真正参与生成**

原因：

- 目前已能存会话，但还没有真正把会话上下文送入问答策略
- 做完这一步，问答体验才会更像助手，而不是“多次单轮提问”
