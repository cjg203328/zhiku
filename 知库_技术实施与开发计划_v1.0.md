# 知库技术实施与开发计划（v1.0）

## 1. 文档目标

本文档承接 `知库_交付版需求_v1.1.md`，输出一份可以直接指导研发开工的实施方案，重点解决以下问题：

- 前端、后端、数据层分别怎么落地
- 哪些模块先做，哪些模块后做
- 为了“可交付”，安装、打包、日志、异常恢复要怎么设计
- 如何把 8 周开发计划拆成可执行任务

本文档默认目标版本为：

- `Zhiku Desktop v1.0`
- 平台：`Windows 10/11 x64`
- 交付形态：本地安装包 + 本地服务 sidecar + 本地数据目录

---

## 2. 交付原则

为保证首版真的能交付，研发实施遵循以下原则：

### 2.1 可安装优先于功能堆叠

如果某项能力会明显增加安装和环境配置成本，应优先考虑降级方案，而不是强行纳入 v1.0。

### 2.2 主链路优先于边角能力

首版必须先打通：

`导入 -> 解析 -> 摘要/标签 -> 入库 -> 搜索 -> 问答 -> 导出/备份`

### 2.3 本地可控优先于外部依赖

- 用户内容默认不出本机
- 关键数据结构和索引由本地维护
- 外部依赖仅限可替换的模型服务或平台解析接口

### 2.4 交付可排障

任何影响主流程的问题，必须能通过：

- 明确的错误提示
- 本地日志
- 可导出的诊断信息

快速定位，而不是只能靠开发者远程猜测。

### 2.5 研发过程可追踪

从当前仓库协作方式出发，后续研发还需要补一层过程约束：

- 只要落地了代码、脚本、配置或正式文档修改，就要同步更新 `docs/product/知库_开发变更日志.md`
- `docs/product/知库_开发进度同步.md` 只维护阶段判断与高层路线，不承担逐次流水账职责
- 本地整理到“可提交”状态前，优先运行 `scripts/dev/pre_push_check.ps1`
- 远程仓库推送默认由用户手动执行，代理不自动 `push`

## 3. 建议代码仓结构

建议使用单仓多模块结构，便于统一版本、打包和发布。

```text
zhiku/
├─ apps/
│  └─ desktop/                    # Tauri + React 桌面端
├─ services/
│  └─ api/                        # Python FastAPI 本地服务
├─ packages/
│  ├─ shared-types/               # 前后端共享类型定义（可选）
│  └─ ui/                         # 通用 UI 组件（可选）
├─ scripts/
│  ├─ dev/                        # 本地开发脚本
│  ├─ build/                      # 构建与打包脚本
│  └─ release/                    # 发布脚本
├─ docs/
│  ├─ product/
│  ├─ architecture/
│  └─ release/
└─ assets/
   ├─ icons/
   └─ templates/
```

### 3.1 模块职责

- `apps/desktop`：负责 UI、窗口、设置页、导入交互、日志入口、安装后的用户体验
- `services/api`：负责解析、文件处理、AI 摘要、向量检索、问答、备份、索引维护
- `scripts`：负责一键启动开发环境、构建 sidecar、生成安装包

## 4. 总体技术方案

## 4.1 运行时架构

建议采用：

- 前端：`Tauri + React + TypeScript`
- 本地服务：`Python + FastAPI`
- 服务运行方式：由桌面端启动本机 sidecar 进程
- 桌面端与服务通信：本机 `HTTP + SSE`
- 数据存储：`SQLite + FTS5 + FAISS + 文件系统`

### 4.1.1 为什么是这个方案

- `Tauri` 打包体积更小，更适合桌面交付
- `Python` 在文档解析、OCR、Embedding、RAG 工具链上更成熟
- `HTTP + SSE` 比直接跨语言嵌入更容易调试和排障
- `SQLite + FAISS` 足够支撑首版 1000 到 10000 条内容规模

## 4.2 Sidecar 打包策略

这是“能交付”的关键设计。

### 方案

- 将 `FastAPI` 服务打包为单独的 `zhiku-service.exe`
- 通过 `PyInstaller` 或 `Nuitka` 生成 sidecar 可执行文件
- 由 `Tauri` 在应用启动时拉起 sidecar
- 桌面端检测 sidecar 健康状态后再加载主界面

### 收益

- 用户**不需要单独安装 Python**
- 服务可独立重启，便于排障
- 后续可独立替换服务版本

### 风险控制

- sidecar 启动超时需要给出明显提示
- 启动失败时要展示日志目录和失败原因
- 保留“重新启动服务”按钮

## 4.3 模型接入策略

### 建议方案

- 大模型：`Ollama` 提供本地 LLM 推理
- Embedding：服务层直接加载本地 embedding 模型
- OCR：`PaddleOCR`，如打包复杂度过高可抽象成可替换实现

### v1.0 取舍

- 不把模型权重直接打进安装包，避免包体过大
- 首次启动时检测 `Ollama` 是否安装、默认模型是否可用
- 若模型缺失，进入“模型检查与安装引导”流程

### 必须实现的模型状态信息

- `Ollama` 是否可用
- 默认聊天模型是否已拉取
- Embedding 模型是否已加载
- OCR 模块是否可用

## 4.4 本地网络约束

- sidecar 仅监听 `127.0.0.1`
- 不暴露到局域网
- 使用固定端口优先，如冲突则自动切换并回传桌面端
- 端口、PID、启动时间写入运行时状态文件

## 5. 核心流程设计

## 5.1 启动流程

1. 启动桌面应用
2. 检测配置目录与知识库目录
3. 拉起 `zhiku-service.exe`
4. 轮询 `GET /api/v1/health`
5. 检测模型状态与数据库状态
6. 若首次启动则进入引导页
7. 正常进入知识库首页

### 失败分支

- sidecar 启动失败：提示“本地服务启动失败”并展示日志入口
- 数据库打开失败：提示检查数据目录权限
- 模型缺失：进入引导页，不阻塞基础浏览能力

## 5.2 导入流程（B 站）

1. 用户粘贴链接
2. 前端调用 `POST /api/v1/imports/url`
3. 服务创建 `import_job`
4. 执行 URL 识别与平台分流
5. B 站解析器提取标题、作者、字幕等信息
6. 生成预览数据
7. 前端展示预览，用户确认保存
8. 服务写入 `contents`、`assets`、`chunks`、向量索引
9. 返回详情页

## 5.3 导入流程（本地文件）

1. 用户拖拽或选择文件
2. 前端调用 `POST /api/v1/imports/file`
3. 服务根据类型进入解析器
4. 抽取文本正文
5. 生成摘要和标签
6. 写入知识库并建立索引

## 5.4 问答流程

1. 用户输入问题
2. 服务做 query embedding
3. 从 `FAISS` 取 Top-K chunk
4. 用元数据与 chunk 构建 prompt
5. 调用本地 LLM 生成答案
6. 组织 citations 返回前端
7. 前端展示可点击引用

- 显示标题、来源、平台、导入时间、摘要、标签
- 展示原文内容或抽取文本
- 展示引用定位锚点
- 支持重新摘要、重新打标签、导出 Markdown、删除

### 6.2.6 AI 问答面板

- 输入问题
- 流式显示回答
- 展示引用来源列表
- 支持点击来源打开内容详情页

### 6.2.7 设置页

- 知识库目录
- 默认模型状态
- 日志目录
- 导出目录
- 备份按钮

### 6.2.8 回收站页

- 查看已删除内容

## 9. 关键实现细节

## 9.1 B 站解析器

### 首版要求

- 支持 `b23.tv`、标准视频 URL、纯 `BV` 号
- 提取标题、作者、封面、时长、播放量、标签、字幕
- 记录失败原因

### 设计建议

- 将平台解析实现为 `parser interface`
- 首版只有 `bilibili_parser`
- 后续新增 `xiaohongshu_parser`、`douyin_parser` 时复用统一接口

### 返回对象统一结构

```json
{
  "platform": "bilibili",
  "title": "...",
  "author": "...",
  "content_text": "...",
  "tags": ["..."],
  "metadata": {},
  "assets": []
}
```

## 9.2 文本切块策略

建议首版采用简单稳定策略：

- 目标 chunk 大小：`500-800` 中文字符
- overlap：`80-120` 字符
- 优先按自然段切分
- 对超长段落再二次切分

## 9.3 摘要输出结构

摘要服务输出必须结构化，至少包含：

- `summary`
- `key_points`
- `quotes`
- `content_type`
- `use_case`

## 9.4 引用格式

问答返回至少包含：

- `answer`
- `citations`
- `session_id`

其中 `citations` 每条至少包含：

- `content_id`
- `title`
- `snippet`
- `chunk_index`
- `score`

## 9.5 日志策略

建议日志拆成三类：

- `app.log`：桌面端事件与启动错误
- `service.log`：sidecar 接口与处理错误
- `import.log`：导入任务详细过程

### 日志要求

- 每次导入和问答生成唯一 `trace_id`
- 错误日志带堆栈
- 默认不记录完整正文，仅记录摘要字段和对象 ID

## 14. 当前最推荐的开工顺序

如果现在马上进入开发，我建议按以下顺序推进：

1. 先搭 `Tauri` 壳和 `FastAPI` sidecar
2. 先把 `SQLite + contents/import_jobs` 跑通
3. 先做 `B站 + DOCX/TXT/MD` 导入
4. 再接摘要与标签
5. 再做搜索与详情页
6. 最后做 RAG 问答、备份、打包

原因很简单：

- 没有稳定入库链路，就没有真实知识库
- 没有真实知识库，RAG 问答只能停留在演示状态
- 没有 sidecar 打包策略，就很难做成真正可交付的 App
